# SOSE Midterm Project 2025/2026: Ethics as a Service

**Author:** Marco Spada  
**Application Domain:** University Services (Course Assignment)

## Project Description
This project implements a small Service-Oriented Architecture (SOA) integrating the concepts of **Data as a Service (DaaS)** and **Ethics as a Service (EaaS)**.
The application simulates a university system where the assignment of a course to a student is dynamically evaluated by an independent ethical engine. This engine makes decisions based on external policies (fairness, accessibility, workload sustainability) and returns a justified and traceable verdict (Audit Trail).

---

## Architecture & Technologies
The system is fully containerized and orchestrated via Docker Compose. It consists of 4 microservices:
1. **Triplestore (Fuseki):** Semantic database exposing the RDF dataset via SPARQL endpoints.
2. **DaaS (Data as a Service):** Java service (Apache CXF, Jena) exposing university data through secure REST APIs.
3. **EaaS (Ethics as a Service):** A *stateless* rule engine in Java that evaluates proposals using dynamic expressions (Apache Commons JEXL) based on external JSON policies.
4. **Client Orchestrator:** A standalone Java script simulating the frontend. It fetches data from the DaaS, normalizes the payload, sends it to the EaaS, and prints the final verdict.

## Build & Execution Instructions

### Prerequisites
- Docker and Docker Compose installed on your machine.

### Starting the Infrastructure
Navigate to the root directory of the project and run:

```bash
docker compose up --build
```

Note: The infrastructure implements native healthchecks. The DaaS will wait for Fuseki to be ready, and the Client will wait for both DaaS and EaaS to be fully operational before proceeding.

## Demo Instructions

Once the infrastructure is up, the `client-app` container will automatically run a complete simulation:

1. Fetches student `S001` (Alice, a working student requiring accessibility) from the DaaS.
2. Fetches course `C001` (Algorithms, high workload, physical presence required) from the DaaS.
3. Sends the proposal to the EaaS.
4. Prints the final Audit Trail (decision, risk level, rationale) directly to the console.

To view the server-side Audit Trail persistence (generated by the EaaS):

```bash
docker logs eaas-service
```

To shut down and clean up the infrastructure:

```bash
docker compose down -v
```

## DaaS (Data as a Service) Documentation
### RDF Dataset (`dataset.ttl`)

The database is modeled using Turtle (RDF) with standard `foaf:` prefixes (for people) and a custom ex: vocabulary (`http://example.org/uni/`).

- Main Classes: `foaf:Person` (Student), `ex:Course` (Course).
- Student Properties: `ex:studentId`, `foaf:name`, `ex:isWorkingStudent`, `ex:needsAccessibility`, `ex:gpa`.
- Course Properties: `ex:courseId`, `ex:title`, `ex:workload` (High/Low), `ex:requiresPhysicalPresence`.


### REST API Endpoints
The DaaS service runs on port `8081` (internally `daas:8080/api`) and exposes the following 5 endpoints:

| Method | Endpoint | Description |
| --- | --- | --- |
| `GET` | `/api/courses` | Returns the complete list of courses. |
| `GET` | `/api/courses/{id}` | Fetches a specific course (e.g., `C001`). Returns 404 if not found. |
| `GET` | `/api/students` | Returns the complete list of students. |
| `GET` | `/api/students/{id}` | Fetches a specific student (e.g., `S001`). Returns 404 if not found. |
| `GET` | `/api/courses/search` | Advanced search (e.g., `?workload=High&physical=true`). Combines multiple filters. |


All endpoints natively return application/json data and are protected by input validation to prevent SPARQL injection.

### EaaS (Ethics as a Service) Documentation
The EaaS service runs on port 8082 and exposes the following endpoint: 
 - `POST /api/evaluate`: Receives a JSON containing student and course. Returns a structured Audit Trail (`auditId`, `timestamp`, `decision`, `riskLevel`, `rationale`).

### Policy Management
The ethical engine does not contain hardcoded rules. Policies are defined in the external file `eaas/policies/corsi_policies.json`, which is mounted at runtime into the container.
The system uses an Expression Evaluator (JEXL) to parse the `condition` field at runtime (e.g., `student.gpa < 20.0 && course.workload == 'High'`), ensuring total decoupling between the software domain and ethical governance.
